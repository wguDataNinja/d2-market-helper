# Data Products

## 0. `rune_registry.json` (Active)

**Purpose:** Canonical registry of all 33 D2R runes with ordering, tier classification, and name crosswalk across all data products. Consumed by the website UI for rune table sorting, segment selector ordering, and cash price matching.

**Status:** Active. `data/rune_registry.json`.

**Schema:** Array of rune objects:
```json
[
  {"id": 1, "name": "El Rune", "short_name": "El", "tier": "low", "names": {
    "in_game": "El",
    "traderie_tools": "El Rune",
    "cash": "El",
    "cash_slug": "el_rune"
  }}
]
```

| Field | Type | Description |
|---|---|---|
| `id` | int | Rune level (1-33) |
| `name` | string | Full display name |
| `short_name` | string | Short identifier without " Rune" suffix |
| `tier` | string | `low` (El-Hel), `medium` (Io-Gul), `high` (Vex-Zod) |
| `names.in_game` | string | Key used in `in_game_rune_values.json` segments.{slug}.runes |
| `names.traderie_tools` | string | Key used in `traderie_tools_prices.json` segments.{slug} |
| `names.cash` | string | `item_name` value in `external_cash_prices.sample.json` observations |
| `names.cash_slug` | string | `item_slug` value in `external_cash_prices.sample.json` observations |

**Tier classification:**
- **low:** El (1) through Hel (15)
- **medium:** Io (16) through Gul (25)
- **high:** Vex (26) through Zod (33)

**Name crosswalk:** The three price products use different naming conventions. `in_game_rune_values.json` keys are `short_name` (e.g. `"Ber"`). `traderie_tools_prices.json` keys are `"{short_name} Rune"` (e.g. `"Ber Rune"`). `external_cash_prices.sample.json` uses `short_name` for `item_name` and `snake_case` with `_rune` suffix for `item_slug` (e.g. `"ber_rune"`). The `names` crosswalk in each registry entry maps between these conventions so the website can join data across products.

## 2. `external_cash_prices.sample.json` (Active)

**Purpose:** Cash listing prices from RMT/marketplace sites. Comparison-only display. Never used for in-game pricing.

**Status:** Active — multi-source product merging IGGM (30), ItemNow (42), D2Stock (199), and items7 (0). 271 total observations, all `use_in_model=false`, schema v0.2.

### Schema

**File:** `data/products/external_cash_prices.sample.json`

**Schema version:** `0.2` (multi-source with normalized fields)

### Required observation fields

| Field | Type | Description |
|---|---|---|
| `source_slug` | string | Source identifier matching `data/source_manifest.json` |
| `evidence_class` | string | Must be `"cash_listing"` |
| `item_name` | string | Display name of the item (e.g. `"Ber"`, `"Zod"`) |
| `normalized_item_name` | string | Trimmed/stripped version of `item_name` |
| `item_type` | string | One of `"rune"`, `"bundle"`, `"item"`, `"unknown"` |
| `price_usd` | number | Cash price in USD (float, e.g. `8.99`) |
| `price_cents` | integer | Cash price in cents (int, e.g. `899`) |
| `currency` | string | ISO currency code, always `"USD"` |
| `segment_confidence` | string | One of `"low"`, `"medium"`, `"high"` |
| `use_in_model` | boolean | **Must be `false`** — cash observations are never in-model |
| `captured_at` | string | ISO 8601 timestamp of capture |
| `item_slug` | string | Snake-case slug (e.g. `"ber_rune"`) |
| `raw_text` | string | Raw text as extracted from source |
| `parser_notes` | string | Notes on extraction method and caveats |

### Optional observation fields

| Field | Type | Description |
|---|---|---|
| `source_url` | string or null | Site-level URL for the source marketplace |
| `product_url` | string or null | Per-item deep-link URL if available |
| `source_artifact_path` | string or null | Path to the capture artifact |
| `quantity` | integer or null | Listed quantity |
| `unit_price` | number or null | Per-unit price |
| `platform` | string or null | Game platform (e.g. `"pc"`) |
| `ladder` | boolean or null | Ladder flag |
| `hardcore` | boolean or null | Hardcore mode flag |
| `softcore` | boolean or null | Softcore mode flag |
| `season_or_ruleset` | string or null | Season identifier (e.g. `"ROTW"`) |
| `caveats` | array of strings | Per-source caveats |

### Multi-source structure

Each per-source file (`data/external/{source}_cash_prices.json`) feeds into the generator which normalizes fields and produces the merged product. The `sources` array at the product level tracks which sources contributed and their observation counts.

### Rules

- **`use_in_model` must always be `false`** — hard-enforced by validator
- **`segment_confidence` defaults to `"low"`** unless the source provides confirmed segment metadata
- **`evidence_class` must be `"cash_listing"`** — the validator rejects any other value
- **Prices must be in USD** — the product only handles cash prices in USD
- **`source_url` and `product_url` are optional** — site-level URLs are acceptable where per-item deep links are unavailable
- **`normalized_item_name` is required** — provides a clean item name for cross-source matching

### Current sources

| Source | Observations | Segment Confidence | URL Type |
|---|---|---|---|
| IGGM | 30 | high — confirmed PC Non-Ladder Softcore ROTW | Site-level only |
| ItemNow | 42 (33 individual + 9 bundles) | low — base prices, not segment-specific | Site-level only |
| D2Stock | 199 (66 individual + 133 bundles) | low — segment labels visible but filter behavior unverified | Product-level |
| items7 | 0 (browser capture required) | — | — |

**Total: 271 observations across 4 sources.**

Source-specific caveats: IGGM segment is fully confirmed (PC / Non-Ladder / Softcore / ROTW). ItemNow items have segment UI filters but base prices may not reflect filtered prices. D2Stock shows segment labels (Ladder vs Non-Ladder) in listing titles with price differences observed (e.g. Ber $7.94 ladder vs $6.84 non-ladder) but filter URL behavior is unverified. items7 requires a rendered browser capture to expose client-side loaded prices.

## Freshness & History

All product outputs carry metadata describing when they were generated, what time window their data represents, and caveats about how that window is understood.

### Fields

| Field | Type | Description |
|---|---|---|
| `product_generated_at` | string (ISO 8601) | Timestamp when the product JSON was generated. Matches `generated_at`. |
| `source_window_label` | string | Describes what time window the source data covers. See possible values below. |

### `source_window_label` Values

| Value | Description | Applies To |
|---|---|---|
| `current_snapshot` | One-time parser run. The data is a point-in-time snapshot, not a historical time series. | `external_cash_prices` |
| `rolling_recent_trades_50_cap` | Traderie returns at most 50 recent completed trades per item/segment. The window is count-capped (50 listings), not time-capped. Historical depth depends on scheduled snapshot retention. | `in_game_rune_values` |
| `unknown_window` | The window behavior of the source is not fully understood. The data may represent a recent window, not full history. | Deprecated — replaced by `rolling_recent_trades_50_cap` for Traderie |
| `historical_window` | Future. Used when the source window is well-characterized (e.g. "last 7 days"). | Not yet assigned |

### Per-Source Window Caveats

**External Cash Prices (`external_cash_prices.sample.json`):**
- `caveat_history`: "Project history starts when snapshots began. Prices are current snapshots, not historical time series."
- These are one-off browser captures or API probes, not continuously collected time series.
- Historical depth begins on 2026-06-20 when the snapshot pipeline was established.

**In-Game Rune Values (`in_game_rune_values.json`):**
- `caveat_window`: "Traderie completed-trade API returns at most 50 recent completed listings per item/segment. Not a full historical archive. Project history depends on scheduled polling and deduped snapshot retention."
- `caveat_pagination`: "nextPage is a boolean/repeating cursor, not a sequential page index. No deeper API pagination observed beyond 50 listings."
- `caveat_history`: "Project history for this source starts when scheduled snapshot retention began. Pre-snapshot history is not available."
- The Traderie API returns at most 50 completed trades per item/segment. This is a count-capped window, not time-capped. Historical depth depends on scheduled snapshots.

### Snapshot History Accumulation

As the snapshot pipeline matures (see `research/memos/2026-06-20-snapshot-history-plan.md`), historical JSONL files will accumulate under `data/history/`. Each run appends new observations with dedup keys. Over time, this enables:

- Trend analysis on cash prices
- Change detection on rune values
- Diff reports across generator runs

The `source_window_label` will be updated per source as the window behavior becomes better characterized. A label of `historical_window` indicates that the source's time window is well understood and the data can be treated as a proper historical series.
